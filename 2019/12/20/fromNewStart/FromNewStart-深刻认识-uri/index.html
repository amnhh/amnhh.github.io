开始

URI 和 URL 的区别？

首先来看 URI， URI 全称 Uniform Resource Identifier, 我们分开来看这三个单词：

  • Uniform: 本意为一致性的 => 规定统一的格式可以方便的处理不同类型的资源
  • Resource: 本意为资源 => 资源的定义，是可以标识的互联网上的任何东西
  • Identifier: 本意就是标识符了

串联起来其实就是：统一资源标识符，URI 其实就像是网上的地址信息一样，在世界范围内唯一标识、定位信息资源

而 URL, 中文名则是统一资源定位符，URL 描述的是一台特定服务器上某个资源的特定位置。可以明确的说明如何从一个精确的、固定的位置获取资源

URI 的格式

主体格式就是：

  • scheme: 方案名。说明的是资源所使用的协议类型
  • auth:user: 用于认证相关的用户名
  • auth:pwd: 用于认证相关的密码
  • host: 主机名，服务器在网络上的地址。可以是域名，也可以直接是公网 ip
  • port: 端口号，默认端口号为 80
  • path: 带层次的文件路径，类似于文件路径
  • search: 查询字符串，这里可以用于 get 请求的参数携带
  • hash: 片段标识符，标识的是文档中的某个位置，一般对应的是文档内的某个标签的 id 属性对应的值

URI 的保留字

  • %: ASCII 25, 百分号用来表示转义符
    • 例如 encodeURIComponent('+') 会返回 %2B
    • % 本身也会被转义，所以需要格外注意重复转义的情况：encodeURIComponent('%') => %25
  • /: ASCII 2F, 斜杠用来表示 path 的层级关系
    • 同样的 . 和 .. 也是保留字
  • #: ASCII 23, hash 符也为保留字，标记的是文档中的某个位置
  • ?: 问号用来当做 URI 和查询对象的分界，例如 a.com?again=1
    • + 常常被当做是空格的一个缩写，所以我们想要真实的 + 携带给后端的话，需要提前将 + 转义
    • 我们通常理解的 get 请求的参数，会被携带到 query 上，也就是 ? 后
