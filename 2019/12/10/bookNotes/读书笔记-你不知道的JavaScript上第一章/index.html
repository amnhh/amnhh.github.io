作用域的理解？

可以理解成存储变量、查找变量的一个规则，也就是我们平时说的 scope 的概念，感觉就像是一个一个把变量画起来的圈

编译性语言还是解释性语言？

JS 其实是一种编译语言，编译语言就会经历以下的步骤（或者说是一个编译器的工作过程？）：

  1. 分词/词法分析
  • Token：词法单元，可以理解成代码的最小单元
  • 这个过程将源代码字符串拆分成一个个 Token
  • 我目前的仓库 stone 也正处在这个阶段
  1. 解析/语法分析
  • 抽象语法树的生成 => AST
  • babel 的 parser 做的事情就在这个阶段
  • 拿到 AST 之后我们就可以做一个节点的替换，这也就是 babel 插件生效的那个阶段
  1. 代码生成
  • AST => Code 的这个过程
  • 此时的 Code 是要引擎可以解析执行的

理解引擎、编译器、作用域三者的职责

  • 引擎： 从头到尾负责整个 JS 的编译及执行
  • 编译器：负责词法分析、语法分析和最后的可执行代码生成
  • 作用域：维护和存储标识符，明确代码对每个标识符的访问权限 => 刚刚我提到的圈子~

LHS 和 RHS？

Left Hand Side 和 Right Hand Side 的简称？

  • LHS：
    • 直意来看是变量出现在等号左手边的时候，会对变量执行 LHS
    • 此时为的是将这个变量在内存中的容器的地址拿出来，并为其做赋值之类的操作
    • 类似 var a = 1 => 此时将 a 拿出来，并拿右边的值为它赋值
    • 可以理解为得到某个变量的盛放容器本身？
    • 赋值的源头
  • RHS:
    • 直意来看就是取到这个值，而并非想把它的容器找到，并赋值
    • 就是简单的查找值
    • console.log(a) => 可以理解成就是拿到现在 a 的值，传给 console.log
    • 取值的源头

博主注：

  • 非左的，就是右？
  • 可以理解成想赋值的时候就用的是 LHS, 想要取值的时候就是 RHS?

为什么区分 LHS 和 RHS？

思考一个例子：

  1. 为什么 x 报错了？
  • 此时我们尝试的去取 x 的值，所以是一个 RHS 的过程
  • 而此时在现有的作用域中找不到这个标识符，所以会抛出异常
  1. 为什么 z 没报错？
  • 我们对 z 的操作，是一个想要为它赋值的过程，所以是 LHS
  • 对 LHS 的过程，看到没有的标识符的时候，只会创建
